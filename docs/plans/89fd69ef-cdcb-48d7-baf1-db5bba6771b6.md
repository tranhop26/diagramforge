### Layout-engine data flow (also documented in README)

```
text ──► parse_spec ──► IR ──► layout.*  ──► Geometry ──► render.svg   ──► SVG string
                              (pure)         (pure)          (pure)
                                                              │
                                                              ▼
                                                         render.mermaid
                                                         render.html
```

`parse_spec`, `layout.*`, and `render.*` are all pure functions with no I/O. The only side-effecting modules are `llm.ts` (network) and the MCP transport (stdio). This separation is what makes determinism cheap to test.

### Determinism rules

- No `Date`, `Math.random`, `process.hrtime`, or environment-derived randomness anywhere under `src/render/`, `src/layout/`, or `src/parsers/dsl.ts`.
- All iteration orders are explicit (sort by id before laying out; sort edges by `(from,to)` for stable emit).
- Theme colors and spacing constants live in `src/themes/presets.ts` and `src/layout/constants.ts` and are imported (not duplicated).
- Test `tests/render-diagram.test.ts` re-renders the same IR 10× and asserts all 10 SVG strings are byte-equal via `Buffer.compare`.

### Error model

- Every tool handler signature is `(args: unknown) => Promise<ToolOutput<T>>`; `tryOk` from `src/tools/_util.ts` is reused unchanged.
- `process.stderr.write(...)` everywhere in `src/`; no `console.log`.
- The unknown-theme fallback logs exactly one stderr line: `[render_diagram] unknown theme "<name>" → light`.
- The LLM fallback path keeps the existing one stderr line: `[parse_spec] LLM failed, falling back to DSL parser`.

### Risk controls

- **SVG well-formedness**: the vendored XML parser in `tests/util/xml.ts` parses every produced SVG. Text content goes through `escapeXml()`. The SVG string is asserted to contain exactly one `<svg` opening tag and exactly one `</svg>` closing tag.
- **Pan/zoom correctness**: implemented against a known-good pattern (translate + scale on a `<g>` wrapper, with `pointer-events` set on the SVG root and `cursor: grab/grabbing` for affordance). Manual offline check is part of AC-9.
- **Mermaid format drift**: format test asserts headers (`flowchart TD`, `sequenceDiagram`, `erDiagram`) and node id regex; documented as a "syntactic sanity check, not a Mermaid-runtime guarantee."
- **ESM/Node16 pitfalls**: every relative import in `src/` ends in `.js`; the `bin` entry uses `./dist/index.js`; `scripts/build-examples.ts` runs via `tsx`; `scripts/check-examples.js` is plain `.js` and imports nothing from `src/` (it reads files only).
- **`npm audit` regressions**: versions are pinned in `package.json`; CI runs `npm audit --audit-level=high`; the `.npmrc` `audit-level=high` is the failure gate.
- **`.gitignore` regression**: the implementor must verify `git check-ignore .github/workflows/ci.yml` reports "not ignored" before committing the CI file.

### Self-dogfood example (committable)

A flowchart whose nodes are `parse_spec` → `IR` → `layout.flowchart` → `render.svg` → `render.mermaid` → `render.html` → `MCP tool: make_diagram`, with an edge from `MCP tool: make_diagram` back to `parse_spec` (the round trip). This is generated by `scripts/build-examples.ts` and lands in `examples/architecture/`. The `examples/architecture/architecture.html` must open offline and show the diagram of DiagramForge's own pipeline.

## Out of Scope

- A state-machine diagram type (the original spec calls it a "nice-to-have bonus, same pipeline" — explicitly omitted; the type system and registry leave a clean extension seam in `src/layout/` and `src/render/`).
- A real Mermaid runtime validator; we ship a syntactic sanity check only.
- Custom user-defined themes (only the three shipped presets are supported; unknown theme → light fallback).
- Authentication, rate limiting, or any HTTP transport (stdio only).
- Persisting state between tool calls; the server is stateless.
- A web UI, hosted demo, or browser extension — only the self-contained HTML preview artifact.
- LLM-driven layout adjustments (the layout engine is intentionally pure and offline; the LLM is only allowed to author IR).
- Pasting finished code from elsewhere. All work happens here through the Humanize implement → review loop.
- A real Mermaid CLI or headless-browser screenshot of `examples/*.html` in CI (manual offline check is the AC-9 evidence).
- Adding a new runtime dependency (zod, `@modelcontextprotocol/sdk`, and `openai` stay as the entire runtime set).
- Any change to the existing `src/types.ts` discriminated union, the existing `src/parsers/dsl.ts` grammar, the existing `src/parsers/prose.ts` routing, the existing `src/llm.ts` OpenAI client, or the existing `src/server.ts`/`src/index.ts` wiring beyond what the new `render_diagram` and `make_diagram` orchestrations require.
