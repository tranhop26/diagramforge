## Assumptions
- The 3 diagram types (flowchart, sequence, ERD) are sufficient to satisfy the "в‰Ґ 3 types" rule; a state-machine implementation is out of scope (see Out of Scope) вЂ” the architecture must still leave a clean extension seam.
- The three shipped themes are `light`, `dark`, `blueprint` (light = default, dark and blueprint also deterministic).
- Sequence message arrows use `->` for sync, `-->` for return; ERD cardinality uses `1--*`, `*--*`, `1--1` style notation.
- `vitest@^1.6.0` and `@modelcontextprotocol/sdk@^1.0.0` are acceptable pin candidates; final versions are picked at scaffold time to satisfy `npm audit в†’ 0 vulnerabilities`.
- The `openai` package is the OpenAI-compatible client referenced by the spec.
- Mermaid output is validated by a syntactic sanity check (matching bracket/quote balance, required keywords) вЂ” not by a real Mermaid runtime, since the server has no Mermaid dependency.
- The four tools live behind a single `createServer()` factory in `src/server.ts`; `src/index.ts` only wires stdio.
- Tests run with `vitest` excluding `examples/` (configured in `vitest.config.ts`).
- Layout collisions are prevented by computed reserved width/height per node and explicit spacing constants, not by overlap-detection passes.
- Examples in `examples/` are generated by a checked-in `scripts/build-examples.ts` (run during the examples task) so reviewers can regenerate deterministically.

### 4. Layout-engine data flow (also documented in README)
```
text в”Ђв”Ђв–є parse_spec в”Ђв”Ђв–є IR в”Ђв”Ђв–є layout.*  в”Ђв”Ђв–є Geometry в”Ђв”Ђв–є render.svg   в”Ђв”Ђв–є SVG string
                              (pure)         (pure)          (pure)
                                                              в”‚
                                                              в–ј
                                                         render.mermaid
                                                         render.html
```
`parse_spec`, `layout.*`, and `render.*` are all pure functions with no I/O. The only side-effecting modules are `llm.ts` (network) and the MCP transport (stdio). This separation is what makes determinism cheap to test.

### 5. Determinism rules
- No `Date`, `Math.random`, `process.hrtime`, or environment-derived randomness anywhere under `src/render/`, `src/layout/`, or `src/parsers/dsl.ts`.
- All iteration orders are explicit (sort by id before laying out).
- Theme colors and spacing constants are exported from `src/themes/presets.ts` and `src/layout/constants.ts`.
- Test `determinism.test.ts` re-renders the same IR 10Г— and asserts all 10 SVG strings are byte-equal via `Buffer.compare`.

### 6. Error model
- Every tool handler signature is `(args: unknown) => Promise<ToolOutput<T>>`.
- A small `tryOk<T>(fn)` wrapper in `src/tools/_util.ts` catches all thrown errors, prefixes the message with the tool name, and returns `{ ok: false, error }`.
- `process.stderr.write(...)` is used everywhere; no `console.log` in `src/`.
- The LLM fallback path logs exactly one stderr line: `[parse_spec] LLM failed, falling back to DSL parser: <reason>`.

### 7. Risk controls
- **SVG well-formedness**: a small vendored XML parser (no dependency) lives in `tests/util/xml.ts`; tests parse every produced SVG. Text content goes through `escapeXml()`.
- **Pan/zoom correctness**: implemented against a known-good pattern (translate + scale on a `<g>` wrapper, with `viewBox` adjustment on resize). Manual offline check is part of AC-9.
- **Mermaid format drift**: format test asserts headers (`flowchart TD`, `sequenceDiagram`, `erDiagram`) and node id regex; documented as a "syntactic sanity check, not a Mermaid-runtime guarantee."
- **ESM/Node16 pitfalls**: every relative import ends in `.js`; the `bin` entry uses `./dist/index.js`; `tsx` is used for the `dev` script and `probe-tools.js` runs the **built** artifact, not source.
- **`npm audit` regressions**: versions are pinned in `package.json`; CI runs `npm audit --audit-level=high`; the `audit-level=high` is the failure gate.

### 8. Self-dogfood example (committable)
A flowchart whose nodes are `parse_spec` в†’ `IR` в†’ `layout.flowchart` в†’ `render.svg` в†’ `render.mermaid` в†’ `render.html` в†’ `MCP tool: make_diagram`, with an edge from `MCP tool: make_diagram` back to `parse_spec` (the round trip).
